## init and halt!

### `ig/init`

Starts the system from a configuration map. Keys are initialized in topological (dependency) order. Returns the **system map** with config values replaced by initialized values.

```clojure
;; Start all keys
(def system (ig/init config))

;; Start only selected keys (and their transitive dependencies)
(def system (ig/init config [:adapter/jetty]))
```

### `ig/halt!`

Stops the system in **reverse** dependency order. Entirely side-effectful — ignore the return value. Discard the system after calling `halt!`.

```clojure
(ig/halt! system)

;; Halt only selected keys
(ig/halt! system [:adapter/jetty])
```

`halt-key!` should be **idempotent** — calling it twice on the same key must be safe.

## suspend! and resume

Use these during interactive development to rebuild parts of the system without closing expensive resources.

### `ig/suspend!`

Like `halt!`, but calls `suspend-key!` instead of `halt-key!`. Intended to pause the system while keeping long-lived resources (connections, open ports) alive.

```clojure
(ig/suspend! system)

;; Suspend only selected keys
(ig/suspend! system [:adapter/jetty])
```

Falls back to `halt-key!` for any key that doesn't define `suspend-key!`.

### `ig/resume`

Like `init`, but takes an additional argument: the suspended system. Calls `resume-key` for keys that implement it, otherwise falls back to `init-key`.

```clojure
(def new-system (ig/resume config system))

;; Resume only selected keys
(def new-system (ig/resume config system [:adapter/jetty]))
```

`resume-key` receives `[key new-opts old-opts old-impl]` — use this to decide whether to reuse the existing resource or reinitialize from scratch.

**Typical REPL workflow:**

```clojure
;; On code reload:
(ig/suspend! system)
;; ... reload namespaces ...
(def system (ig/resume new-config system))
```

See [workflows.md](workflows.md) for the full integrant-repl workflow.

## expand

The `expand` function runs `expand-key` on every key in the configuration that has a method defined, then deep-merges the results into a new configuration. Call `expand` before `init`.

```clojure
;; Basic usage
(-> config ig/expand ig/init)

;; With an inner processing function (e.g. for profiles)
(-> config (ig/expand #(ig/deprofile % [:dev])) ig/init)

;; Concise form using deprofile as a function
(-> config (ig/expand (ig/deprofile [:dev])) ig/init)
```

Values set **directly** in the configuration take priority over those generated by `expand-key`. This allows overriding individual keys from a module:

```edn
{:module/greet {:name "Alice"}
 :adapter/jetty {:port 3000}}   ; overrides the port set by :module/greet's expansion
```

## converge

`converge` is a lower-level primitive that migrates a running system toward a target configuration, performing the minimum necessary halts and inits.

```clojure
(def new-system (ig/converge system new-config))

;; Converge only selected keys
(def new-system (ig/converge system new-config [:adapter/jetty]))
```

Keys that haven't changed are left running. Keys removed from config are halted. New keys are initialized. Changed keys are halted and reinitialised. This is similar to `suspend!/resume` but driven entirely by config diffing.

## Profiles

Profiles let you vary parts of the configuration for different environments without maintaining separate config files.

### `#ig/profile` tagged literal

Embed a profile map inside any config value. Each key is a profile name; the value for the matching profile is selected.

```edn
{:adapter/jetty {:port #ig/profile {:dev 8080, :prod 80}}}
```

### `ig/deprofile`

Resolves all `#ig/profile` tags in a configuration, selecting values for the given profiles in order. The first matching profile key wins.

```clojure
;; Select :dev profile
(ig/deprofile config [:dev])
;; => {:adapter/jetty {:port 8080}}

;; Try :test, fall back to :dev
(ig/deprofile config [:test :dev])
```

Raises an exception if a profile tag contains no key matching any supplied profile.

### Profiles inside expansions

`expand-key` can return a profile map; use the inner function form of `expand` so that `deprofile` runs before expansions are merged:

```clojure
(defmethod ig/expand-key :module/greet [_ {:keys [name]}]
  (ig/profile
   :dev  {:adapter/jetty {:port 8080} :handler/debug {:name name}}
   :prod {:adapter/jetty {:port 80}   :handler/greet {:name name}}))

;; Correct order: deprofile happens before merging
(ig/expand config (ig/deprofile [:dev]))
```

## Vars

A **var** is a placeholder in the configuration for a value that will be supplied at runtime. Use vars to keep secrets or environment-specific values out of the config EDN.

```edn
{:adapter/jetty {:port #ig/var port}}
```

```clojure
;; Bind values before init
(def bound-config (ig/bind config {'port 8080}))
(ig/init bound-config)
```

Calling `ig/init` on a config that still contains unbound vars throws an exception.
