## Configuration Maps

A configuration map is a plain Clojure map (or EDN file) where each top-level key identifies a component and its value is the component's configuration:

```clojure
(require '[integrant.core :as ig])

(def config
  {:adapter/jetty  {:port 8080, :handler (ig/ref :handler/greet)}
   :handler/greet  {:name "Alice"}})
```

Or as pure EDN (loaded with `ig/read-string`):

```edn
{:adapter/jetty  {:port 8080, :handler #ig/ref :handler/greet}
 :handler/greet  {:name "Alice"}}
```

```clojure
(def config (ig/read-string (slurp "config.edn")))
```

**Rules:**
- Every top-level key must be a **qualified keyword** (namespace required) — Integrant validates this
- Values can be any EDN-readable data, plus `#ig/ref`, `#ig/refset`, `#ig/var`, and `#ig/profile` tagged literals
- Keys may also be **vectors of qualified keywords** (composite keys — see [advanced-features.md](advanced-features.md))

## Multimethods

Integrant dispatches on the key. Every multimethod takes at least the key and its value/implementation as arguments.

### `init-key`

Called when the system starts. Takes the key and its config value; returns the initialized value that replaces the config value in the system map.

```clojure
(defmethod ig/init-key :adapter/jetty [_ {:keys [handler port]}]
  (jetty/run-jetty handler {:port port :join? false}))

(defmethod ig/init-key :handler/greet [_ {:keys [name]}]
  (fn [_] (resp/response (str "Hello " name))))
```

**Initializer function shorthand:** if no specific method is defined for a key, Integrant looks for a function with the same namespace and name as the keyword:

```clojure
(def config {::sugared-greet {:name "Alice"}})

(defn sugared-greet [{:keys [name]}]   ; same ns + name as ::sugared-greet
  (println "Hi" name))

(ig/init config)   ; calls sugared-greet automatically
```

The namespace containing the function must be loaded first (use `ig/load-namespaces`).

---

### `halt-key!`

Called when the system stops. Takes the key and the **initialized** value (the return value of `init-key`). Should be side-effectful and **idempotent**. Return value is ignored.

```clojure
(defmethod ig/halt-key! :adapter/jetty [_ server]
  (.stop server))
```

Not every key needs a `halt-key!` — if there is nothing to clean up, omit it.

---

### `suspend-key!`

Called during development when rebuilding the system without closing long-lived resources (database connections, open ports). Signature mirrors `halt-key!`. Return value is ignored; must be idempotent.

```clojure
(defmethod ig/suspend-key! :adapter/jetty [_ {:keys [handler]}]
  (reset! handler (promise)))   ; block new requests without stopping the server
```

Falls back to `halt-key!` if not defined.

---

### `resume-key`

Called after `suspend-key!` to bring the system back up. Takes four arguments: key, new config value, old config value, old initialized value.

```clojure
(defmethod ig/resume-key :adapter/jetty [key opts old-opts old-impl]
  (if (= (dissoc opts :handler) (dissoc old-opts :handler))
    (do (deliver @(:handler old-impl) (:handler opts))
        old-impl)
    (do (ig/halt-key! key old-impl)
        (ig/init-key key opts))))
```

Falls back to `init-key` if not defined (ignores the suspended system).

---

### `resolve-key`

Applied to a value **before** it is resolved as a reference. Use this to hide internal implementation details from dependent keys. Takes the key and its initialized value; returns the resolved value that dependents see.

```clojure
(defmethod ig/resolve-key :adapter/jetty [_ {:keys [server]}]
  server)   ; dependents get the Jetty server object, not the map
```

---

### `expand-key`

Called during the `expand` phase (before `init`). Acts like a macro: takes the key and config value, returns a map that is deep-merged into the configuration. Used to build reusable modules.

```clojure
(defmethod ig/expand-key :module/greet [_ {:keys [name]}]
  {:adapter/jetty {:port 8080, :handler (ig/ref :handler/greet)}
   :handler/greet {:name name}})
```

Values set directly in the configuration take priority over those generated by expansions.

---

### `assert-key`

Called after `init-key` to validate the initialized value. Throw (or use `assert`) to signal an invalid system state.

```clojure
(defmethod ig/assert-key :adapter/jetty [_ {:keys [port]}]
  (assert (nat-int? port) ":port must be a valid port number"))
```

Errors are wrapped in `ExceptionInfo` with `:reason :integrant.core/build-failed-spec`.

## Refs and Refsets

### `ig/ref`

A reference to exactly one key in the configuration. Resolved before `init-key` is called. The referee must be unambiguous (only one matching key).

```clojure
;; In code
{:adapter/jetty {:handler (ig/ref :handler/greet)}}

;; In EDN
{:adapter/jetty {:handler #ig/ref :handler/greet}}
```

The ref does not need to match the exact key — it can match a parent in a derived hierarchy (see [advanced-features.md](advanced-features.md)).

### `ig/refset`

A reference to **all** keys derived from the given keyword. The resolved value is a set of all matching initialized values.

```clojure
;; In EDN
{:handler/greet-all {:names #ig/refset :const/name}
 :const.name/alice  {:name "Alice"}
 :const.name/bob    {:name "Bob"}}
```

```clojure
(defmethod ig/init-key :handler/greet-all [_ {:keys [names]}]
  (fn [_] (resp/response (str "Hello " (str/join ", " names)))))
```

## The System Map

After `ig/init`, the returned system map has the same top-level keys as the configuration, but each value is replaced by its initialized value:

```clojure
;; config
{:adapter/jetty {:port 8080, :handler #ig/ref :handler/greet}
 :handler/greet {:name "Alice"}}

;; system (after init)
{:adapter/jetty <Jetty server object>
 :handler/greet <ring handler fn>}
```

Pass the system to `ig/halt!` to stop it. **Do not hold references to config values** after init — work with the system map.

## Namespace Conventions for Keys

Integrant keys must be **qualified keywords**. Namespace choice is yours, but two common conventions exist:

| Style | Example | When to use |
|-------|---------|-------------|
| `component/name` | `:adapter/jetty` | Infrastructure components shared across the app |
| `my.ns/component` | `::db` (expands to `my.ns/db`) | Component defined in and local to a specific namespace |

The `load-namespaces` function tries to load the namespace part of each key, so naming keys after the namespace that defines their multimethods enables auto-loading.
